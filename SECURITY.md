# PolyWeather Security Implementation

## Overview

This document describes the comprehensive security implementation for the PolyWeather trading bot, addressing all critical vulnerabilities identified in Freya's security audit.

## ⚠️ CRITICAL FIXES IMPLEMENTED

### 1. Private Key Security ✅
**Issue**: Plain text private keys in .env files  
**Fix**: Enterprise-grade encrypted key storage

- **Implementation**: AES-256-CBC encryption with PBKDF2 key derivation
- **Key Storage**: `~/.polyweather/keys/` with 0600 permissions
- **Features**:
  - Automatic migration from plaintext environment variables
  - Key rotation capabilities
  - HMAC integrity verification
  - Master password protection

**Usage**:
```python
from polyweather.security.key_manager import get_secure_private_key

# Automatically decrypts and returns key
private_key = get_secure_private_key("polygon_private_key")
```

### 2. TOTP Emergency Authentication ✅
**Issue**: Hardcoded emergency token "***"  
**Fix**: RFC 6238 TOTP-based authentication

- **Implementation**: Time-based One-Time Passwords with backup codes
- **Features**:
  - QR code generation for easy setup
  - 10 backup codes for emergency access
  - 30-second time windows with ±30s tolerance
  - Secure secret generation and storage

**Usage**:
```python
from polyweather.security.totp_auth import verify_emergency_token

# Verify 6-digit TOTP or 8-character backup code
if verify_emergency_token(user_token):
    # Grant emergency access
    execute_emergency_action()
```

### 3. WebSocket Security ✅
**Issue**: No authentication, no rate limiting  
**Fix**: JWT authentication + comprehensive rate limiting

- **Features**:
  - JWT-based client authentication
  - Per-client rate limiting (100 req/min default)
  - Burst attack detection (10 req/5s limit)
  - IP-based connection limits (5 per IP)
  - Permission-based channel access
  - TOTP required for emergency channels

**Authentication Flow**:
1. Client connects to WebSocket
2. Server sends authentication challenge
3. Client sends JWT token
4. Server verifies token and grants access
5. All messages are rate-limited per client

### 4. API Security ✅
**Issue**: CORS allows all origins, no rate limiting  
**Fix**: Restricted CORS + comprehensive rate limiting

- **CORS Configuration**:
  - Allowlist specific origins only
  - Restricted HTTP methods
  - Secure headers configuration
  - 10-minute cache for preflight requests

- **Rate Limiting**:
  - 100 requests per minute per IP
  - 20 burst requests allowed
  - Sliding window algorithm
  - Emergency endpoints: 10 req/5min

### 5. Redis Security ✅
**Issue**: No authentication, plaintext sensitive data  
**Fix**: Authentication + encryption for sensitive data

- **Features**:
  - Redis password authentication
  - AES encryption for sensitive categories
  - Encrypted data categories: positions, signals, config, trading, wallet
  - Automatic encryption/decryption
  - Secure connection handling

## 🔧 Setup Instructions

### 1. Install Security Dependencies
```bash
cd polyweather-bot
pip install -r requirements-security.txt
```

### 2. Run Security Setup Script
```bash
python scripts/setup_security.py
```

This script will:
- Generate encryption keys
- Set up TOTP authentication
- Migrate plaintext keys
- Validate security configuration

### 3. Configure Environment Variables

**Remove from .env** (now stored encrypted):
- `POLYMARKET_API_KEY` 
- `OPENWEATHER_API_KEY`
- `POLYGON_PRIVATE_KEY`

**Add to .env**:
```bash
# Security keys (generated by setup script)
POLYWEATHER_MASTER_KEY=<generated_master_key>
POLYWEATHER_JWT_SECRET=<generated_jwt_secret>
POLYWEATHER_CACHE_KEY=<generated_cache_key>

# Redis security
REDIS_PASSWORD=<secure_redis_password>

# API security  
CORS_ALLOWED_ORIGINS=http://localhost:3000,https://dashboard.polyweather.local
```

### 4. Configure TOTP Authentication

After running setup:
1. Scan QR code with authenticator app (Google Authenticator, Authy, etc.)
2. Store backup codes securely
3. Test authentication:
```bash
python scripts/test_emergency_auth.py
```

### 5. Update Redis Configuration

Add to `redis.conf`:
```
requirepass <REDIS_PASSWORD>
```

Restart Redis to apply authentication.

## 🔒 Security Features

### Key Management
- **Encryption**: AES-256-CBC with PBKDF2-SHA256 (100,000 iterations)
- **Storage**: Encrypted files with 0600 permissions
- **Key Derivation**: Master password + salt
- **Integrity**: HMAC-SHA256 verification

### Authentication
- **TOTP**: RFC 6238 compliant, 30-second windows
- **JWT**: HS256 signature, 1-hour expiry
- **Backup Codes**: 8-character alphanumeric, single-use

### Rate Limiting
- **Algorithm**: Sliding window with burst detection
- **Granularity**: Per-IP and per-client tracking
- **Adaptive**: Load-based threshold adjustment
- **Emergency**: Separate strict limits for emergency endpoints

### Data Protection
- **Redis**: Category-based encryption for sensitive data
- **Transmission**: HTTPS/WSS for all communications
- **Storage**: Encrypted at rest for all credentials

## 🚨 Emergency Procedures

### Emergency Trading Controls
All emergency controls now require TOTP authentication:
```bash
# Emergency stop with TOTP token
curl -X POST http://localhost:8080/api/v1/emergency/control \
  -H "Content-Type: application/json" \
  -d '{"action": "liquidate_all", "confirm_token": "123456"}'
```

### Key Recovery
If master key is lost:
1. Stop all trading immediately
2. Regenerate API keys from providers
3. Run security setup script again
4. Update configuration with new keys

### TOTP Recovery
If TOTP device is lost:
1. Use backup codes for emergency access
2. Generate new TOTP secret
3. Update authenticator app
4. Generate new backup codes
